# StandupDigest

Purpose: For report compilers (PMs, eng managers, ops analysts) who owe a weekly stakeholder status — drop the Jira/Linear/Asana/GitHub issue CSV you already export and get a categorized, copy-ready weekly status digest, 100% in the browser.

Problem: Report COMPILERS — PMs, engineering managers, ops/data analysts, team leads — who owe a WEEKLY stakeholder status. They already export the CSV from their tracker; the recurring grunt work is sorting it by status, grouping by person/epic, spotting carry-over, and writing prose. A PM writing a weekly stakeholder status from an Asana/Jira export is the literal north star: recurring (weekly), single-session value, 30-second legible.

Beats alternative: Best alternative today = manually sorting the export in a spreadsheet (pivot/sort/group, then hand-write the prose) ~20–30 min/week, or pasting it into an LLM (leaks tracker data, still needs cleanup/formatting). StandupDigest turns it into a ~30-second drop-the-CSV → categorized digest, saving ~20–30 min/week per compiler with ZERO data leaving the browser. The paid incumbents (Standuply, Friday.app, Geekbot, eazyBI) all require connecting your tracker or a per-seat plan; the no-integration / no-upload / no-signup, fully client-side categorization is the reason this app exists.

Core flows (≤4; copy AND share are shared sub-actions of every view, not standalone flows):

SHARED SHARE CONTRACT (applies to every view below, like copy): after a digest renders (Weekly Status, Sprint Review, or Changes), the user may click "Create shareable link" to publish a read-only snapshot of the CURRENTLY-FORMATTED digest now on screen. Before/at creation the UI explicitly discloses WHAT is uploaded — only the formatted digest text shown (prose summary + categorized issue titles, assignees, epics, points, transition tags) — vs WHAT stays local and is NEVER uploaded (the raw CSV, unmapped rows, Backlog/Todo rows not in the digest body, and the column mappings in localStorage). One write to Turso mints a secret unguessable URL; opening that URL in any fresh context (no login) renders the SAME formatted view READ-ONLY: no dropzone, no re-upload, no column mapping, no inline editing, no "Create link" button — purely the formatted digest plus a "Made free with StandupDigest — no signup" footer that links to the app. The shared view is mobile-glanceable. One read loads it; no polling, no accounts. The link's "Copy link" button obeys the same in-place confirmation contract as Copy (below): the cue renders on the persistent trigger button, flips optimistically to a solid-green "Link copied ✓" with aria-live, and falls back to a hidden textarea when the clipboard is blocked.
1. Drop or select a Jira/Linear/Asana/GitHub issue CSV export (or click "Load sample data") → the file is parsed in-browser (never uploaded), columns are auto-detected via the tracker header-alias map, each row is bucketed by status keyword into Shipped / In Progress / Blocked (Backlog/Todo kept separate from the digest body), rows are grouped by assignee AND by epic, carry-over/spillover rows are flagged, and an editable weekly digest with a one-line prose summary renders. A WEEK FILTER selector at the top of the digest filters which rows appear by their Updated/Resolved date and DEFAULTS to the most recent ISO week present in the data (labeled "Week of <date>"), with an always-available "All dates" option; in a selected week, SHIPPED/resolved rows are filtered to that week but still-open (In Progress/Blocked) rows whose last update predates the week REMAIN and are flagged carry-over. Every parsed row lands in exactly one bucket; any status that matches no keyword surfaces in a visible "Unmapped status" list rather than being dropped. SHARED COPY CONTRACT (applies to this and every view below): each view offers "Copy as Markdown" and "Copy as plaintext"; the clicked button flips IN PLACE to "Copied ✓", the copied text EXACTLY matches the on-screen content, and the flip fires optimistically with a hidden-textarea fallback when the clipboard is blocked.
2. Fix a misdetected column via a manual "map your columns" picker (the mapping is remembered in localStorage per detected source: jira/linear/asana/github) OR edit any digest/changes line inline (commit on Enter AND blur AND explicit Save into the same model used for display + copy) OR re-bucket an entry from the Unmapped list into Shipped/In Progress/Blocked/Backlog.
3. Toggle to a "Sprint Review" tab (Weekly Status stays the default; the same already-parsed CSV is reused) → using the same column auto-detect/manual-mapping/localStorage plus two added columns (Story Points/Estimate, Sprint/Cycle), a sprint selector defaulting to the most recent sprint drives an editable sprint-review block showing VELOCITY (points completed vs committed, plus issue counts), SCOPE CHANGE (points/issues added or removed mid-sprint, or an explicit "scope change unavailable" note when the data can't support it), SPILLOVER (issues still open at sprint end, reusing the carry-over logic), and BY ASSIGNEE (points + issue counts per person), under the shared copy contract.
4. Toggle to a "Changes" tab (Weekly Status stays the default; the current CSV is reused as the "now" side) and OPTIONALLY load a PRIOR-period export in a second dropzone labeled "Compare to last week's export (optional)" (or click "Load sample data" to load BOTH a current and prior-week sample at once) → both files are bucketed with the SAME status→bucket map and rows are matched across exports by a stable id (Issue Key / ID / URL / permalink), falling back to exact Title with a visible "matched by title (less reliable)" note when no id column exists. A SINGLE computed changes model classifies every matched/new row into exactly ONE transition bucket — Newly Shipped, Newly Started, Newly Blocked, Unblocked, Slipped (In Progress/Blocked prior → Backlog/Todo now, i.e. lost momentum), Reopened (Shipped prior → not-Shipped now, i.e. came back from done), New this period, Still Blocked (with "blocked 2+ wks" only when the prior date supports it), Carried over/unchanged-open — plus a "Removed from tracker" list for prior-only rows. Slipped and Reopened are SEPARATE buckets with separate headings and counts. An editable one-line prose summary ("Since last week: N shipped, N newly blocked, N slipped, N reopened, N new") and the per-bucket tags and the copied Markdown/plaintext ALL read their counts from that ONE model (never recomputed). Headline buckets (Newly Shipped/Blocked/Slipped/Reopened/New) lead; Carried-over and Still-Blocked are secondary/collapsible. Same-file-twice → "No changes detected — are these the same export?"; clearly different trackers → an amber unreliable-match warning but the diff still renders.

Header-alias auto-detect map (builder MUST implement exactly; case-insensitive match):
- Title/Summary — Jira `Summary`; Linear `Title`; Asana `Name`; GitHub `title`.
- Status — Jira `Status`; Linear `Status`/`State`; Asana `Section/Column` (+ `Completed At`); GitHub `state` (open/closed) + Projects `Status`.
- Assignee — Jira `Assignee`; Linear `Assignee`; Asana `Assignee`; GitHub `assignee`/`assignees`.
- Epic/Group — Jira `Epic Link`/`Epic Name`/`Parent`; Linear `Project`/`Cycle`; Asana `Projects`/`Parent task`; GitHub `milestone`/`labels`.
- Updated/Resolved date — Jira `Updated`/`Resolved`; Linear `Updated`/`Completed`; Asana `Completed At`/`Modified`; GitHub `updated_at`/`closed_at`.
- Story Points/Estimate (numeric; Sprint Review tab) — Jira `Story Points`/`Story point estimate`/`Custom field (Story Points)`; Linear `Estimate`; Asana numeric `Story Points`/`Points` custom field; GitHub a `points`/`estimate` label or column.
- Sprint/Cycle (Sprint Review tab) — Jira `Sprint`; Linear `Cycle`; Asana `Section`/custom field; GitHub `milestone`/`iteration`.

Status→bucket keyword map (builder MUST implement exactly; case-insensitive):
- Shipped: Done, Closed, Resolved, Complete, Completed, Merged, Released; GitHub `closed`; Asana `Completed At` non-empty.
- In Progress: In Progress, In Review, In Dev, Started, Doing, Active, QA.
- Blocked: Blocked, On Hold, Waiting, Stalled, Impediment.
- Backlog/Todo (kept separate, NOT in the digest body): To Do, Backlog, Open, New, Triage. GitHub `open` defaults here unless a Projects Status says otherwise.
- Reconciliation rule: every parsed row lands in exactly one bucket; any status matching no keyword goes to the "Unmapped status" list (never silently dropped).
- Spillover: a row whose status is unchanged across the export's Updated window (or carried from a prior cycle field) is flagged as carry-over.

Out of scope:
- Anything needing accounts, tracker integration, or CSV file upload to a server. The default flow (no share link created) stays fully client-side — the CSV never leaves the browser. The ONLY server interaction is the opt-in share-link write/read, which sends ONLY the already-formatted digest text the user chose to share (see SHARED SHARE CONTRACT); the raw CSV is never sent under any flow.
- Burndown CHARTS / graphical visualizations (bars, lines, sparklines) — the Sprint Review tab reports velocity/scope/spillover/by-assignee as NUMBERS and text only; charts are the separate SprintLens idea, explicitly out of scope here.
- The WEEK FILTER remains a single-week filter only (not a delta). Cross-export comparison now lives in the dedicated, OPTIONAL "Changes" tab, which requires the user to load a second prior-period export — single-file Weekly Status never auto-computes a week-over-week delta on its own.
- Editing the source CSV (you edit the digest, not the input file).
- Persistence of the RAW CSV / mappings across devices (localStorage holds only the per-source column mapping; the raw CSV and unmapped/Backlog rows are NEVER sent anywhere). Sharing is now supported ONLY as an explicit opt-in read-only snapshot of the formatted digest text (see SHARED SHARE CONTRACT) — never automatic, never the raw file.

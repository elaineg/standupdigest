# StandupDigest — UX Brief

## 1. Problem statement
Drop the export your project tracker already gives you, and instantly get a publish-ready weekly status update you can paste into Slack — free, with nothing ever leaving your computer.

## 2. Primary user action
Drop (or pick) a Jira/Linear/Asana/GitHub issue CSV onto a single large dropzone and watch a categorized weekly digest render below it. The user never starts at a blank box: the dropzone has a prominent **"Load sample data"** button right inside it that seeds a known-value example digest in one click, so a skeptic sees the finished outcome before touching their own file.

> **Round-2 correctness principle (builder fix, stated here):** the copied output — BOTH Markdown and plain text — must EXACTLY match what is on screen: no phantom assignee `(Bob)` on rows the UI shows with no assignee, and the chosen Assignee/Epic grouping must be reflected in the copied text (not silently flattened).

## 3. Emotional tone
Calm, trustworthy, businesslike — like a clean internal tool a manager already trusts. Neutral cool grays + one confident blue accent; green reserved for "Shipped" and copy-confirmation only. System/sans font (Inter-feel), comfortable but efficient spacing — never cramped, never playful.

## 4. Design decisions
- **Show, don't tell — instant populated example.** The cold screen is never a lonely empty dropzone: a single-click "Load sample data" inside the dropzone fills the full digest (5 Shipped / 4 In Progress / 2 Blocked / 3 Backlog) so the value is legible in seconds. A "Clear" link returns to empty.
- **Copy confirmation lives on the button, unmissably.** Each copy button flips IN PLACE to a solid-green "Copied ✓" with `aria-live="polite"`, holds ~2s, then restores. The flip is optimistic (fires before the clipboard promise) and falls back to selecting a hidden textarea if clipboard is blocked — so a hurried human always sees it worked. No sub-second toast.
- **Distinct verbs on adjacent controls.** The two copy buttons read **"Copy Markdown"** and **"Copy plain text"** (different trailing words, never two "Copy" pills that look like one broken control). Inline digest editing = a pencil "Edit line"; unmapped re-bucketing = a "Move to…" dropdown; column fixes = "Remap columns" — three different verbs, never adjacent duplicates.
- **No-silent-drop trust signal, sized to its content.** The "Unmapped status" list always renders so users trust nothing was dropped, but when empty it collapses to one quiet line ("All statuses recognized ✓"); when populated it shows each unknown status with a "Move to…" control.
- **Column mapping is always reachable and never falsely reassuring (Wen, Marcus).** A persistent **"Remap columns"** link sits in the digest header (next to the week label) at ALL times, regardless of detection confidence — a user whose Title/Status/Assignee was mis-detected can always reopen the mapping dialog. The amber low-confidence banner + auto-open of the dialog triggers when ANY core field (Title, Status, OR Assignee) fails to detect — not only on overall low confidence. Three trusted states: (1) **fully detected** → quiet "All statuses recognized ✓" success line; (2) **partial / low-confidence** (a core column unmapped, e.g. titles fell to "(untitled)") → amber banner + auto-opened mapping dialog, and the "✓" success line is SUPPRESSED (never show success while a core column is unmapped — it gives false reassurance); (3) always, in any state → the manual "Remap columns" entry point in the header.

## 5. Three screen-states
**A. Cold / empty (above the fold, incl. 375px):**
- Headline: *"Turn your tracker export into a weekly status — in seconds."*
- Subtitle: *"Drop a Jira, Linear, Asana, or GitHub CSV. Get a Shipped / In Progress / Blocked digest ready to paste into Slack."*
- One large dropzone with **[Load sample data]** button inside it (the focal CTA when no file is loaded).
- Privacy line directly under the dropzone, always visible: *"Your file never leaves your browser — no upload, no signup."*

**B. Populated digest:** A **digest header row** carries the **week selector** (left) and the persistent **"Remap columns"** link (right). Directly below: an **editable** prose one-liner reading **"Week of <date>: the team shipped 5 items, has 4 in progress and 2 blocked, with 1 carried over."** — click to edit inline like the digest lines (Sam); the edited text flows verbatim into BOTH copy outputs. Then **Shipped / In Progress / Blocked** sections (Backlog/Todo shown separately, visually muted), a **Group by: Assignee ▸ Epic** toggle, carry-over rows tagged, the collapsed-when-empty Unmapped list, and a sticky **[Copy Markdown] [Copy plain text]** pair. Inline "Edit line" pencils per item.

- **Week filter (D — Elena, Dana, Sam).** Above the buckets, a single **week / date-range selector** filters which rows appear by their Updated/Resolved date, DEFAULTING to the most recent ISO week present in the data, labeled clearly (e.g. **"Week of Mon 9 Jun – Sun 15 Jun ▾"**). Changing it re-renders the buckets and updates the "Week of <date>:" prose stamp. This is a single-week FILTER ONLY — NOT week-over-week comparison/delta (that stays out of scope). On 375px the selector sits full-width at the top of the digest, above the prose line, stacking above the buckets.
- **Copy bar is a STATIC non-overlapping footer (A-fix — Marcus, Jules, Aisha, Rob).** The "Copy Markdown / Copy plain text" button bar is a STATIC footer at the bottom of each digest card — it flows in the normal document below the last row, never sticky or fixed to the viewport, so it can NEVER float over or cover digest rows mid-list. Background is solid white opaque. On Sprint Review (where pb-24 bottom-padding remains), the last by-assignee row always clears the bar. Both copy buttons are persistent (never unmount) so the in-place "Copied ✓" confirmation is always visible. Verified at 1280px and 375px: zero overlap with any digest row on Weekly Status, Sprint Review, and Changes tabs.
- **Copy confirmation visible on persistent button (D — Rob).** After the non-overlapping A-fix, both Copy Markdown and Copy plain text flip in-place to solid-green **"Copied ✓"** with `aria-live="polite"` for ~2s, then restore. The button never unmounts on click so the cue is always visible. Confirmed with both normal clipboard and blocked-clipboard (textarea fallback) paths.
- **Share link is the PRIMARY action (B — Aisha).** The "Share link" button is a FILLED BLUE PILL (bg-blue-600, font-semibold, with a link-glyph) — visually dominant relative to both the Copy buttons and the "Remap columns" text link. Its label contains "Share" + a link icon; it cannot be mistaken for a third copy variant or a peer of "Remap columns." It is discoverable as the headline stakeholder-sharing action.
- **Column mapping + status rules SAVED on this device (C-i — Wen, Tomás, Dana, Elena).** A persistent note reading **"Saved on this device — next week just drop your new export."** is visible near the "Remap columns" entry point in the digest header. After applying a mapping in the Remap panel, a "Saved on this device — next week just drop your new export." confirmation appears inline. Custom unmapped-status→bucket rules are also persisted per source in localStorage: once a user re-buckets an unknown status (e.g. "Needs Triage Review" → In Progress), that rule is saved and auto-applied on the next load of the same source — so next week's export maps custom statuses without re-fiddling.

**C. Low-confidence / partial column mapping:** When any core field (Title, Status, OR Assignee) fails to auto-detect, the digest still renders best-guess but the mapping dialog AUTO-OPENS and an amber banner appears above it — *"We weren't sure which columns are which. [Remap columns]"* — with per-field dropdowns (Title / Status / Assignee / Epic / Date). The "All statuses recognized ✓" success line is suppressed in this state. The choice is remembered in localStorage per source. The "Remap columns" link in the digest header is present in ALL states; only the auto-open + amber banner are conditional.

## 5-second check (what a cold visitor sees above the fold)
Headline ("Turn your tracker export into a weekly status — in seconds"), one-line subtitle naming the four trackers + the Slack-paste outcome, the dropzone with a **Load sample data** button, and the privacy line — so within 5 seconds a PM/EM reads "drop my export, get a status to paste, free, nothing uploaded." On 375px the headline, subtitle, dropzone, Load-sample button, and privacy line all stay above the fold; the digest stacks vertically below. Above the dropzone, the two-mode tab strip (Weekly Status | Sprint Review) is also visible cold (see Sprint Review tab) so a PM/EM instantly sees there are two outputs from the same drop.

## 6. Sprint Review tab (additive — Weekly Status stays the default, UNCHANGED)
A PM/EM can produce a sprint-review readout from the SAME CSV they already drop — velocity, scope change, spillover, by-assignee — without re-importing or learning a new app.

- **Tab/toggle placement & affordance.** A single horizontal **tab strip sits at the very top of the page content, directly under the "StandupDigest" header bar and ABOVE the hero/dropzone** — visible in BOTH cold and populated states. Two segmented tabs reusing the existing toggle look (same blue-100 / blue-700 active pill as the "Group by: Assignee ▸ Epic" control): **[ Weekly Status ]  [ Sprint Review ]**. **Weekly Status is the default-active tab on every load** (active = filled blue pill; inactive = gray text). Switching tabs swaps ONLY the body below; the dropzone, parsed rows, column mapping, and the Clear / Load-file controls are shared — drop once, flip tabs freely. The active tab persists in localStorage (`standupdigest-mode`) but always defaults to Weekly Status if unset. Cold state shows the same hero/dropzone under either tab, with the subtitle line adapting under Sprint Review to: *"Drop the same CSV — get velocity, scope change, spillover, and points by assignee for a sprint."*
- **Sprint Review layout (reuses the white rounded-2xl card, same section rhythm as the digest).** A **digest-style header row** carries the **sprint selector** (left) and the persistent **"Remap columns"** link (right), identical placement to the week selector. Below it, in order:
  - **VELOCITY headline** — the focal number, large and bold like the prose one-liner: **"42 of 55 points shipped"** (completed Shipped-bucket points vs total committed points in the selected sprint), with a quiet sub-line **"18 of 23 issues done"**. Completed number in green (matching the Shipped accent); committed in gray.
  - **SCOPE CHANGE** — a one-line stat: **"+8 pts / 3 issues added, −2 pts / 1 issue removed mid-sprint."** When the data can't support it, show the exact muted fallback **"Scope change unavailable — needs a Sprint + Added-date column"** (never a wrong number), styled like the muted Backlog note.
  - **SPILLOVER / CARRY-OVER** — reuses the existing carry-over logic and the amber `carry-over` tag look: **"Spillover: 4 issues / 13 pts still open at sprint end,"** then the list of those issues (same EditableLine rows).
  - **BY ASSIGNEE** — reuses the GroupedSection assignee grouping, each group header showing the person plus their points + issue count, e.g. **"👤 Sam — 13 pts · 4 issues."**
- **Sprint selector.** Same affordance and position as the week filter: a labeled dropdown **"Sprint:"** at the left of the card header, listing the sprints/cycles detected in the data (most recent first) and **defaulting to the most recent sprint present**, labeled by its sprint name (e.g. **"Sprint 24 ▾"**). Changing it recomputes velocity/scope/spillover/by-assignee for that sprint.
- **Editable + copy controls.** The whole sprint-review block is **editable inline** (same pencil "Edit line" pattern; the velocity headline and each stat line are editable). The SAME sticky opaque bottom bar carries **[Copy Markdown] [Copy plain text]**, with the SAME copy-fidelity contract: copied text EXACTLY matches the on-screen sprint review (headline, scope, spillover list, by-assignee), and the clicked button flips in place to solid-green **"Copied ✓"** for ~2s with `aria-live`.
- **Mobile (375px).** Tab strip stays full-width and pinned at top (two equal-width tabs). The sprint selector sits full-width above the velocity headline; velocity stays glanceable (the big "X of Y points shipped" line is the first thing visible on the card). Sticky copy bar keeps its opaque bg + scroll padding so by-assignee rows clear it.
- **5-second-rule note.** Because the tab strip renders cold above the dropzone, a PM/EM sees TWO labeled outputs ("Weekly Status", "Sprint Review") before doing anything; tapping Sprint Review then Load-sample shows a populated velocity headline ("42 of 55 points shipped") within seconds — the value of the second mode is legible without their own data.

## 7. Changes since last week (additive — Weekly Status stays the default, UNCHANGED)
A PM/EM can ALSO drop LAST week's export of the same tracker and get a "what moved" digest — newly shipped, newly blocked, slipped, new — without diffing two spreadsheets by hand. Single-file Weekly Status is untouched; this is a third tab.

- **Tab strip & second dropzone.** The top tab strip becomes three segmented tabs reusing the same blue-pill look: **[ Weekly Status ]  [ Sprint Review ]  [ Changes ]**. Weekly Status stays default-active on every load. The Changes body has the SAME current-file dropzone PLUS a SECOND, visually paired drop zone beneath it labeled **"Compare to last week's export (optional)"** with helper text *"the previous CSV export of this same tracker."* The current file (already loaded under any tab) is reused as the "now" side; only the prior file is new to load here. A single **"Load sample data"** button on this tab loads BOTH a current AND a prior-week sample at once, so one click renders a full Changes digest.
- **Empty Changes state (constraint 4 — 5s legibility).** Before a prior file exists, the body shows a short worked explanation, not a blank panel: a one-line title *"See what changed since your last export."*, a one-line subtitle *"Drop last week's CSV in the second box — we'll show what shipped, what's newly blocked, and what slipped."*, the second dropzone, and the **[Load sample data]** button (loads both samples). A cold PM understands within 5s that this compares against a PRIOR file and exactly where to put it.
- **Prose summary (placement & count honesty, constraint 1).** Directly under the card header, the editable one-liner: **"Since last week: 4 shipped, 2 newly blocked, 1 slipped, 3 new."** Counts are read from the SINGLE computed changes model (see §8) — never recomputed for the prose. Editing it flows verbatim into both copy outputs.
- **Transition-group visual hierarchy.** Headline buckets first, in this order, each a section like the digest's Shipped/In-Progress/Blocked, with the SAME row + assignee + inline-pencil look: **Newly Shipped** (green accent), **Newly Blocked** (red/amber accent), **Slipped / Reopened** (amber), **New this period**, **Unblocked**, **Newly Started**. SECONDARY, visually muted and COLLAPSIBLE (collapsed by default): **Carried over / unchanged-open** and **Still Blocked** (the Still-Blocked rows show "blocked 2+ wks" only when the prior export's date supports it, else "still blocked"). A small muted **"Removed from tracker"** list renders last (rows only in the prior export). Every matched/new row appears in EXACTLY ONE bucket — no row silently dropped.
- **Match basis note.** When rows are matched by a stable id (Issue Key / ID / URL / permalink) it's silent. When NO id column exists and matching falls back to exact Title, a visible note renders at the top of the Changes body: *"Matched by title (less reliable) — no ID column found."*
- **Edge cases (constraint 5).** Same file dropped as both current and prior → a clear inline note *"No changes detected — are these the same export?"* (never a blank panel). Two clearly different trackers (little/no id overlap) → an amber warning *"These look like different exports — matching may be unreliable"* but the diff still attempts and renders. Missing key column → the title-match reliability note above.
- **Copy bar (constraints 1–3, identical contract).** The SAME sticky opaque bottom bar carries **[Copy Markdown] [Copy plain text]**; copied text is derived from the SAME changes model as the prose counts and on-screen rows (prose number == rows shown == copied text). Inline edits commit on **Enter AND on blur AND on explicit Save** into that one model before any copy. Each button flips in place to solid-green **"Copied ✓"** with `aria-live`, optimistic (fires before the clipboard promise) with a hidden-textarea fallback if the clipboard is blocked. The bar keeps its solid bg + scroll padding so the last bucket (Removed-from-tracker) and editable rows always clear it and Copy stays click-hittable after an edit.
- **Mobile (375px).** Three equal-width tabs pinned at top; the two dropzones stack full-width; headline buckets stack above the collapsible secondary groups; sticky copy bar keeps opaque bg + scroll padding.

## 8. The single changes model (binds constraints 1–2)
All Changes output — prose counts, per-bucket tags, AND copied Markdown/plaintext — derives from ONE computed `changes` object built once from (currentRows, priorRows). The view renders that object; the copy serializers walk that SAME object; the prose counts read its bucket lengths. No count is ever recomputed independently. Inline edits mutate this one model (on Enter/blur/Save) before serialization, so an edited line is identical on screen and in both copy formats.

## 9. Shareable read-only link (additive — opt-in; default flow stays 100% client-side)
A PM/EM who's produced a digest (any of the three views) can publish a secret no-login URL that renders the SAME formatted view read-only, to paste into Slack/email — stakeholders glance at it on their phone without re-running anything. This is OPT-IN and changes nothing until the user clicks. The thing shared is exactly the formatted digest already on screen — never the raw file.

- **PRIVACY TRUST-BOMB — the absolute claim must stay absolute by default (NON-NEGOTIABLE).** This app's wedge is "your tracker data never leaves the browser." A server makes that claim a contradiction the moment a link exists, so the promise is MODE-AWARE:
  - **No link created (the default for everyone, every load):** the existing line renders UNCHANGED and UNQUALIFIED — *"Your file never leaves your browser — no upload, no signup."* No asterisk, no hedge. The cold visitor and every non-sharing user sees the pure promise.
  - **At link creation:** a short disclosure panel appears (inline, above the Create button, or as the confirm step of the same click) that states plainly, in two columns or two lines: **UPLOADED →** *"only the formatted digest you see here — the summary line and the categorized issue titles, assignees, epics"* (and points/transition tags where shown). **STAYS ON YOUR DEVICE →** *"your raw CSV, any Backlog/Todo and unmapped rows, and your column mappings — never uploaded."* A one-line reassurance: *"Anyone with the link can view this digest. Don't create one for confidential data."* The user must see this BEFORE the URL is minted (the first click reveals disclosure + a confirm "Create link"; it does not silently POST).
  - **After a link exists (this session):** the page-level privacy line becomes accurate, e.g. *"Your CSV stays in your browser. You've shared a read-only copy of this digest via link."* — never shown next to the unqualified absolute claim.
- **DISCOVERABILITY — first-class, never confused with Copy (NON-NEGOTIABLE).** Place **"Create shareable link"** in/next to the digest header beside the Remap link, OR as a clearly-set-apart control adjacent to the sticky copy bar — visually distinct from the two copy buttons (e.g. a link-glyph + the word **Share**, set off by a hairline gap or different button style), and its label MUST contain "link"/"Share", never "Copy", so it can't read as a third copy variant. It appears only once a digest is rendered (nothing to share before that). On 375px it stays full-width and visible, not collapsed behind a menu.
- **The shared read-only view (`/v/<id>`).** Pure display: the prose summary + the bucketed sections + tags exactly as the source rendered them. NO dropzone, NO "Load sample data", NO Remap, NO inline-edit pencils, NO week/sprint selector controls, NO Copy/Create-link buttons — strip every input affordance. Mobile-glanceable: single column, big readable prose line first, buckets stacked, no horizontal scroll at 375px (it WILL be opened from Slack on a phone). A footer reads **"Made free with StandupDigest — no signup"** linking to the app home (the viral hook). A tiny meta line may note which view it is (Weekly Status / Sprint Review / Changes) and the week/sprint label, since selectors are gone.
- **COPY-LINK CONFIRMATION (NON-NEGOTIABLE, same friction lesson as Copy).** After minting, show the URL in a selectable read-only field with a **"Copy link"** button. The confirmation MUST render on that PERSISTENT button (it never unmounts on click): it flips optimistically — before the clipboard promise resolves — to a solid-green **"Link copied ✓"** with `aria-live="polite"`, holds ~2s, restores. If the clipboard API is blocked, a hidden-textarea selection fallback still copies and the flip still fires; the field stays selectable so the user can copy by hand regardless. Unmissable green fill, not a sub-second toast.
- **No new top-level surface.** Share is a shared sub-action of the existing three tabs (like Copy), not a fourth tab — the snapshot captures whichever view (Weekly Status / Sprint Review / Changes) and selector state was on screen at create time. Default tab, dropzone, and client-side promise are untouched.
